**Задачи:**
- Изучить методы чтения данных из разных источников (PostgreSQL, Excel, CSV).
- Освоить техники очистки данных: удаление дублей, заполнение пропусков, приведение типов.
- Научиться согласовывать и объединять данные из разных источников через `pd.merge`.
- Реализовать анализ и визуализацию качества технической поддержки.

## 3. ETL-скрипт

Файл: `etl_support.py` — модульная структура, логирование в `etl_support.log`.

**Шаги pipeline:**

1. **`read_data()`** — чтение из трёх источников: `pd.read_sql` (PostgreSQL via SQLAlchemy), `pd.read_excel` (openpyxl), `pd.read_csv`.

2. **`clean_data()`** — для каждого датафрейма:
   - `drop_duplicates()` — удаление дублей;
   - `fillna(0)` для числовых, `fillna('Unknown')` для строковых;
   - `pd.to_datetime()` для колонок с датами (`created_at`, `resolved_at`).

3. **`merge_data()`** — двухэтапное объединение:
   - `support_tickets ⨝ reg_agg` по ключу `category` (добавляет нормативы из Excel);
   - результат `⨝ quality_agg` по ключу `agent_id` (добавляет средний рейтинг агента из CSV);
   - тип объединения `how='outer'` с `indicator=True` для контроля качества;
   - вычисление флага `in_time` — закрыт ли тикет в нормативный срок.

4. **`analyze()`** — расчёт метрик: рейтинг по агентам, среднее время по категориям, % SLA, топ-3 агента, распределение приоритетов.

5. **`visualize()`** — 4 графика matplotlib, сохранение в `plots/`.

6. **`save_to_db()`** — запись итогового датафрейма в таблицу `integrated_support_data` (PostgreSQL, `if_exists='replace'`).

## 6. Выводы

В ходе работы реализован полный ETL-конвейер на Python: данные успешно прочитаны из трёх разнородных источников (PostgreSQL, Excel, CSV) и объединены в единый датафрейм из 120 строк и 15 колонок без потери записей. SLA выполняется лишь в **44.4% случаев** — 4 из 5 категорий нарушают нормативы, наибольшее превышение у Hardware (+6 ч) и General (+4 ч). Лидеры по качеству — агенты AGT003, AGT008, AGT007; аутсайдер — AGT006 (рейтинг 1.60), требует отдельного внимания. Приоритет High составляет 37% обращений, что указывает на высокую нагрузку критических инцидентов. **Рекомендации:** ввести контроль SLA по категориям Hardware и General, провести аттестацию агента AGT006, оптимизировать распределение нагрузки для снижения доли High-приоритетных тикетов.
